1 Tidy Data: Long But Machine Readable

Working with data made me conscious of how we compile, store, and share data. Most of us work with spreadsheets and most of the time we encode our data in a way that makes it easy for other people to read our data. Unfortunately, human readable data format is not necessarily machine readable data. When we want to work with data in code, human readable data is not usually the best form to begin with and to share with other people who wants to reproduce our work.

Wickham (2014) has the following definition of a tidy data:

Data is tidy when

  1. Every column is a variable,
  2. Every row is an observation, and
  3. Every observational unit forms a table.

Data that is not tidy is messy.

Ellis and Leek (2014) also advocates tidy data principles and good practices when sharing data for collaboration.

I wanted to discuss these three properties in the context of a real world data in one post, but the data I will work with is quite messy, not the messiest I have seen, but messy enough for me to work on a lot of things. This post will be the first post in the “From Messy to Tidy” series.

2 Consumer Price Index of the Philippines

The data I will use here is the Philippines consumer price index from 1994 Jan to 2018 Jun retrieved here on 2018 Sep 28. All the tables in this post are displayed using DT::datatable() (Xie (2018)).

Figure 2.1: First 20 Rows of the Philippines Monthly Consumer Price Index (CPI) for All Income Households by Commodity Group and Geographic Area from 1994 Jan to 2018 Jun. CPI from 1994 to 2005 are the equivalent/estimated 2006-based CPI using the splicing method. Data from Philippine Statistics Authority (http://openstat.psa.gov.ph).

According to the Philippine Statistics Authority

Consumer Price Index (CPI) - is an indicator of the change in the average retail prices of a fixed basket of goods and services commonly purchased by households relative to a base year.CPI stands for the percentage change in the average prices of goods and services commonly bought by a group of consumers from the base year.

The initial reason why I looked at this data was because of the high inflation rates the Philippines is experiencing in 2018. However, after I saw how the data looked, I became more interested in using it to discuss tidy data principles and ways to turn a data set from messy to tidy.

3 Messy Data Cues

3.1 Wide not Long

Usually, tidy data takes the form of a long table. This is just a consequence of keeping variables in columns and observations in rows. A wide table, like Figure 2.1, is a common cue that a data is not tidy. The CPI data has 324 columns and 12,095 rows. The dimensions may indicate a long table already, but we must further investigate; 324 columns is just too many to be just variables.

3.2 Dates as Columns

One of the common ways of making a data set messy is by creating columns that are not variables but values of a certain observation, which happens quite often with dates. In the CPI data, there are 294 columns that are month and year data, not even in ISO 8601 format (YYYY-MM-DD). Under each date column is supposed to be the consumer price index for that month and year. Putting variables under variables is not the tidy way.

3.3 Average Columns

Before I do something about these dates, I also noticed that there are Ave_yy columns. These columns show the average CPI for each year.

Ellis and Leek (2014) considers raw data in the correct format if it did not

  1. Ran on software
  2. Modify data
  3. Remove data
  4. Summarise data

I can only assume that the data encoder of the CPI data followed the first three conditions. There are missing values, but I think it was missing in the first place rather than removed. I am sure that the Ave_yy summarizes the data. Manipulating data to obtain any statistical value like the average is easy for any data analyst when the raw data is tidy or at least near tidy. Having summary columns are not only unnecessary, but it also may lead the data analyst into checking what these columns mean1. From the definition of CPI, I also don’t think that the annual average is a necessary and correct value to consider.

4 Short Steps to a Long Table

For this post, I have the following tasks:

  1. Remove the Ave_yy columns,
  2. Collect all the month_year and CPI values underneath into separate columns, and
  3. COnvert the new date columns in ISO 8601 format2.

I opted at separating the code and the discussion of my process such that the discussion will not be interrupted by code blocks and such that the reader can review my process through code in one uninterrupted pass after reading the discussion. For all the packages and code I used, please refer to 7.

4.1 Remove Columns with Names that Starts with the Same Characters

dplyr::select() (Wickham et al. 2018) allows for selecting or removing columns by column names. There are 24 Ave_yy columns. Listing them one-by-one is bothersome.

dplyr::starts_with() (Wickham et al. 2018) offers one way to remove the Ave_yy columns. As the function name implies, dplyr::starts_with() selects all column names that begin with the specified string prefix. We can remove 24 Ave_yy columns using two simple functions. Just don’t forget the - to tell select that we want the columns removed not selected. I also removed the extra and empty X324 column.

4.2 Gather the Date Columns

Converting a table from wide to long format is possible with tidyr::gather() (Wickham and Henry 2018). It takes four important arguments: the wide data, the key or column name we want to call the the column names we are gathering, the value or the column name we want to call the variables under the columns we will collect, and ... or the set of column names we want to gather.

Since the column names we want to gather are in month_year form, it is appropriate to call the “key” column “month_year”. The values under the month_year columns are the CPI for each month; the appropriate “value” column is “cpi”.

There are 294 month_year columns, so listing them one-by-one is more bothersome than listing the Ave_yy columns. dplyr::starts_with() will not help us here because it needs a specific string to match as prefix. The month_year columns have different beginnings (i.e. Jan, Feb, Mar, Apr, May, Jun, Jul, Aug, Sep, Oct, Nov, and Dec). In situations like this, we will need the help of dplyr::matches() (Wickham et al. 2018). Instead of a specific string prefix, dplyr::matches() finds column names that matches a regular expression anywhere in the column name.

The mention of regular expressions might have disturbed the balance of the universe, but nothing to worry, {rebus} (Cotton 2017) is here. With the month.abb (built-in vector of three-letter abbreviations of English month names) and paste(), we can create the regular expression to select all the month_year columns with paste(month.abb %R% "_" %R% DGT %R% DGT, collapes = "|")3. I used collapse = "|" because dplyr::matches() only takes one regular expression as argument, and | is interpreted as the “or”" separator.

Here is the sample of the elongated data so far.

Figure 4.1: The Data so Far. No Ave_yy Columns and month_year Columns Gathered in One Column.

Looking good so far. The data went from wide (324 rows by 12,095 columns) to long (7 rows by 3,555,930 columns) in just a few lines of code. However, the month_year column is still not in ISO 8601 format. This step will be tricky.

4.3 Convert to YYYY-MM-DD

Years in the CPI data are abbreviated to only the last two digits of the year. This makes it very difficult for future encoding of data. What happens when the data goes beyond 2094 or if we obtained data since 1800? This is another reason why ISO 8601 format is preferable, there is no confusion about the year because it is spelled out. However, this is why we are tidying the data, so that future data analysts and gatherers can update the data with consistency and ease.

To manipulate the two-digit years, I first split the month_year column to month and year using tidyr::separate() (Wickham and Henry 2018). With the two-digit year column, I added 1900 when years are in the 90s and 2000 otherwise.

I like working with dates using {lubridate} (Spinu, Grolemund, and Wickham 2018). A “YYYY-MM-DD” string can be converted to a YYYY-MM-DD date with lubridate::ymd(). Since the original CPI data only had months and year, and we can assume that these end-of-month CPI values, I had to arbitrarily add a day field. I used 1 because all months begin with 1; months end at different days, then there is also the leap year. My initial date is ymd(paste(year, month, 1, sep = "-")). To convert this to end-of-month dates, I used lubridate::ceiling_date(), specifying unit = "month" for rounding-up by month. However, lubridate::ceiling_date() rounds up a month to the first day of the next month (see the ?lubridate::ceiling_date() for more information). To obtain the real end-of-month date, I had to subtract a period of 1 day using lubridate::days(1).

Converting to ISO 8601 looks easier in code, but it was a tricky process to get here when you are just thinking about how to handle the messy data for the first time.

7 Code

The following codes show what I did in this post. The logic of each code should have been captured in the discussion, but I added comments here and there to further clarify my thought process.

# Attach packages and set chunk options

library(knitr)
library(tidyverse)
library(magrittr)
library(lubridate)
library(DT)
library(rebus)

opts_chunk$set(
  message = FALSE,
  warning = FALSE,
  echo = FALSE
)


# Read CPI Data

raw_cpi_data <- 
  readr::read_csv(
    "./data/ph-cpi-data/raw/CPI_2006=100_1994-2018_1_2.csv",
    trim_ws = TRUE
  )


# Show Sample of CPI Data

raw_cpi_data %>% 
  head(20) %>% 
  DT::datatable(
    rownames = FALSE,
    extensions = 'FixedColumns',
    options = list(
      scrollX = TRUE,
      scrollCollapse = TRUE
    )
  )


# Remove the Ave_yy columns

no_ave_cpi_data <- 
  raw_cpi_data %>% 
  select(-starts_with("Ave"), -X324)


# Gather the month_year columns

long_cpi_data <- 
  no_ave_cpi_data %>% 
  gather(
    month_year,
    cpi,
    matches(paste(month.abb %R% "_" %R% DGT %R% DGT, collapse = "|"))
  )


# Show CPI data after gathering, so far

long_cpi_data %>%
  head(20) %>% 
  DT::datatable(
    rownames = FALSE,
    extensions = 'FixedColumns',
    options = list(
      scrollX = TRUE,
      scrollCollapse = TRUE
    )
  )


# Convert month_year column into ISO 8601 date column

iso_8601_cpi_data <- 
  long_cpi_data %>% 
  # Split month and year values
  separate(
    month_year,
    into = c("month", "year")
  ) %>% 
  mutate(
    # Convert year to numeric
    year = as.numeric(year),
    # Add 1900 to 90s and 2000 to 2k years
    year = if_else(
      year > 93,
      year + 1900,
      year + 2000
    ),
    # Combine date values and convert to YYYY-MM-DD format; use start of month first
    date_cpi = ymd(
      paste(year, month, 1, sep = "-")
    ),
    # Change to end of month
    date_cpi = ceiling_date(date_cpi, unit = "month") - days(1)
  ) %>% 
  # remove the month and year columns
  select(-month, -year)


# SHow data after converting dates to ISO 8601 format

iso_8601_cpi_data %>%
  head(20) %>% 
  DT::datatable(
    rownames = FALSE,
    extensions = 'FixedColumns',
    options = list(
      scrollX = TRUE,
      scrollCollapse = TRUE
    )
  )


saveRDS(iso_8601_cpi_data, "./data/ph-cpi-data/partial-tidy/iso-8601-cpi-data.RDS")
